## 5. 与原始 pipeline 版本的相同点与差异

## 相同点

1. **核心验证链路一致**
   - 都依赖 `lab_env.Environment` + `llvm_helper`。
   - 都以 `check_fast/check_full` 作为最终验证依据。
2. **结果格式目标一致**
   - 都输出包含 patch、check 结果、日志信息的 JSON。
3. **单 issue 主流程结构一致**
   - 准备 LLVM 工作区 → 构造任务 → 生成补丁 → 验证 → 落盘。

## 差异点

1. **执行框架不同**
   - `agent/run.py`：基于 mini-swe-agent（带轨迹、step/cost 限制、bash tool 调用）。
   - `pipeline/generate.py`：传统脚本式对话/迭代逻辑（以多轮采样和函数 hunk 驱动）。

2. **输出组织方式不同（本次新增）**
   - `agent/run.py`（本次改后）：输出目录与文件统一带时间戳，避免覆盖，便于回溯。
   - `pipeline/generate.py`：默认输出如 `<issue_id>.json`，目录由环境变量控制，通常不强制时间戳。

3. **轨迹可观测性不同**
   - `agent/run.py`：天然产出 `.traj.json`，可直接观察 step_limit/api_calls/exit_status。
   - `pipeline/generate.py`：以 messages/full_messages 聚合日志为主，无 mini-swe-agent 轨迹结构。

4. **停止条件表达不同**
   - `agent/run.py`：显式出现 `LimitsExceeded`（达到 step_limit）。
   - `pipeline/generate.py`：更偏向内部循环次数/采样次数控制。
